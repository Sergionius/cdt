# CDT

CDT is a project-local command line toolkit for Flutter release workflows:
test builds, production builds, web deploys, App Store/TestFlight uploads,
Firebase App Distribution uploads, notifications, and YAML-defined pipelines.

## Features

- Flutter iOS and Android test builds with upload steps.
- Production iOS and Android flows.
- Flutter web deploy to a separate repository.
- Project-local `.env` configuration.
- Optional `cdt.yaml` pipelines with built-in and plugin steps.
- JSON pipeline inspection and validation for automation and agents.

`cdt test --id ...` runs a mixed pipeline for Flutter projects:

1. Increment Flutter build number in `pubspec.yaml` (`version: X.Y.Z+N` -> `X.Y.Z+N+1`)
2. iOS build via Flutter (`flutter build ipa --obfuscate --split-debug-info=obfsymbols`)
3. Android build via Flutter (`flutter build appbundle --obfuscate --split-debug-info=obfsymbols --no-shrink`)
4. Upload built IPA to TestFlight (Apple Transporter)
5. Wait for build processing in App Store Connect API
6. Set TestFlight changelog: `dev build <ids>`
7. Upload AAB to Firebase App Distribution (AppTester) with release notes:
   - one tracker issue URL per `--id`

Uploads start immediately after each platform build completes (no cross-platform wait).

Both iOS and Android builds are started in parallel.

## Required project env (`<target_project>/.env`)

```env
ASC_KEY_ID=your_asc_key_id
ASC_ISSUER_ID=your_asc_issuer_id
ASC_PRIVATE_KEY_PATH=ios/app-store-key.p8
IOS_BUNDLE_ID=com.example.app
ASC_WAIT_TIMEOUT_SEC=30

FIREBASE_APP_ID_ANDROID=your_firebase_android_app_id
FIREBASE_TOKEN=your_firebase_token
FIREBASE_GROUPS=testers

NOTIFY_PROVIDER=telegram
TELEGRAM_BOT_TOKEN=your_telegram_bot_token
TELEGRAM_CHAT_ID=your_telegram_chat_id
NOTIFY_TEXT=Release notification
# for webhook notifications instead of telegram:
# NOTIFY_PROVIDER=pachca
# PACHCA_WEBHOOK_URL=https://example.invalid/webhook

# optional terminal UI mode
# CDT_UI=pretty   # default
# CDT_UI=verbose  # full command logs

# optional success/fail sounds (macOS)
# SUCCESS_SOUND=macos
# SUCCESS_SOUND_FILE=~/Sounds/success.mp3
# FAIL_SOUND=macos
# FAIL_SOUND_FILE=~/Sounds/fail.mp3
# SOUND_VOLUME=0.3
# if *_SOUND_FILE is omitted, system sound is used
# if *_SOUND_FILE is set but file is missing, cdt prints a warning

TRACKER_OAUTH_TOKEN=your_tracker_oauth_token
TRACKER_ORG_ID=your_tracker_org_id
# optional:
# TRACKER_BASE_URL=https://tracker-api.example.invalid

WEB_REPOSITORY=/path/to/web-repository
WEB_BUILD_PLACE=/path/to/web-repository
WEB_INNER=true

# prod user-agent notification
UA_APP_NAME=ExampleApp
PACHCA_USER_AGENT_WEBHOOK_URL=https://example.invalid/user-agent-webhook
# optional user-agent message overrides:
# UA_TITLE=ExampleApp user-agent
# UA_IOS_DEVICE=ios 17.0 iPhone ios-device-id
# UA_ANDROID_DEVICE=android 29 Android android-device-id

# iOS-only commands (xcodebuild archive/export)
IOS_INFO_PLIST=ios/Runner/Info.plist
IOS_TEST_SCHEME=ExampleTestScheme
IOS_PROD_SCHEME=ExampleProdScheme
IOS_CONFIGURATION=Release
# optional override if MARKETING_VERSION can't be resolved via xcodebuild:
# IOS_MARKETING_VERSION=1.2.3
# defaults shown below if not overridden:
# IOS_WORKSPACE=ios/Runner.xcworkspace
# IOS_PROJECT=
# Optional plist path. If missing/invalid, cdt auto-generates temporary ExportOptions plist.
# IOS_EXPORT_OPTIONS_PLIST=ios/ExportOptions.plist
# optional for autogenerated export plist:
# IOS_EXPORT_METHOD=app-store
# IOS_SIGNING_STYLE=automatic
# IOS_TEAM_ID=ABCDE12345
```

Notes:
- `.env` is loaded from the **project you run `cdt` in**, not from the `cdt` repo.
- Keep secrets out of git (`.env` and `.p8` should be ignored).
- `ASC_WAIT_TIMEOUT_SEC` controls how long cdt waits for TestFlight processing (default: `30`).
- Success notification text format:
  - optional header from `NOTIFY_TEXT`
  - `💡 Version: <new build version>`
  - if `--id` provided (`test`/`ios-test`): section `Задачи:` with tracker links.
- Terminal UI mode:
  - `CDT_UI=pretty` (default): compact status-style output
  - `CDT_UI=verbose`: full command-style logs
- Optional success/fail sounds are supported for `test`, `prod`, `ios-test`, `ios-prod`, `deploy`, and `firebase_deploy`:
  - set `SUCCESS_SOUND=macos` and/or `FAIL_SOUND=macos`
  - optional `SUCCESS_SOUND_FILE` / `FAIL_SOUND_FILE` for custom audio file (`.mp3`, `.wav`, `.aiff`, ...)
  - optional `SOUND_VOLUME` (0.0..1.0), default `0.3`
  - default success sound: `/System/Library/Sounds/Glass.aiff`
  - default fail sound: `/System/Library/Sounds/Basso.aiff`
- For each `--id`, cdt attempts to add a Yandex Tracker comment:
  - optional header from `NOTIFY_TEXT`
  - `💡 Версия: <new build version>`
- `cdt firebase_deploy` first runs `flutter build web --release` with the same dev/test flavor as non-prod builds, then executes `firebase deploy`.
- `cdt deploy` does not bump version; it uses current `pubspec.yaml` version as git commit message in `WEB_REPOSITORY`.
- Before deploy starts, cdt force-cleans local git state (`git restore .` + `git clean -fd`) and checks out `main` (fallback: `master`).
- `cdt deploy` appends the current Flutter build number to bootstrap/main JS URLs; with `WEB_INNER=true`, it also renames `index.html` to `index-inner.html` and updates `index-wrapper.html`.
- `cdt prod` with `NOTIFY_PROVIDER=pachca` sends an extra user-agent message to `PACHCA_USER_AGENT_WEBHOOK_URL` using:
  - fixed labels: `iOS:` and `android:`
  - version parts from `pubspec.yaml` (`major` + `build`)
  - app name from `UA_APP_NAME`
- `cdt ios-test` and `cdt ios-prod` are iOS-only, use Xcode archive/export and upload to TestFlight.
- iOS commands are path-driven by env (`IOS_*`) and do not require Flutter `ios/`/`android/` directory checks.
- `cdt ios-test` supports repeated `--id` for Tracker comments; `cdt ios-prod` runs without `--id`.
- `ios-test`/`ios-prod` bump iOS build number in `Info.plist` (`IOS_INFO_PLIST`), not in `pubspec.yaml`.
- Legacy `NATIVE_*` iOS env names are still accepted for existing projects.

## iOS Xcode build requirements

`ios-test`/`ios-prod` use `xcodebuild -exportArchive`.
If `IOS_EXPORT_OPTIONS_PLIST` is not set (or points to missing file), cdt auto-generates
`build/ios/<scheme>/ExportOptions.auto.plist`.

You can tune generated export options with:
- `IOS_EXPORT_METHOD` (default: `app-store`)
- `IOS_SIGNING_STYLE` (default: `automatic`)
- `IOS_TEAM_ID` (optional)

## Prerequisites

`cdt` uses only Flutter + Apple tooling + Firebase CLI (no Ruby/Fastlane for iOS upload).

Required tools:

```sh
flutter --version
xcrun iTMSTransporter -help
firebase --version
```

If tooling is missing, `cdt test` will stop with a clear error message.

### Install Firebase CLI (if missing)

```sh
npm install -g firebase-tools
# verify
firebase --version
```

Build bump example:
- `version: 1.1.22+419` -> `version: 1.1.22+420`

## Usage

```sh
# TEST (default: both iOS + Android)
cdt test --id exampl_1 --id exampl_2
cdt test --only ios --id exampl_1
cdt test --only android --id exampl_1

# PROD (no --id, changelog is fixed: "prod build")
cdt prod
cdt prod --only ios
cdt prod --only android
# when both platforms are enabled, iOS and Android start in parallel;
# Android builds run sequentially: AAB first, then APK

# iOS-only flows (xcodebuild archive/export, fastlane gym analog)
cdt ios-test --id exampl_1 --id exampl_2
cdt ios-prod

# Firebase Hosting deploy (builds web with dev/test params first)
cdt firebase_deploy

# Web deploy to repository + git push
cdt deploy
```

## YAML pipelines

`cdt.yaml` lets projects define trusted pipelines with built-in CDT steps and
project-local Python plugin steps. Plugins are trusted project-local Python
code: CDT imports them into the current process before validating or running a
pipeline, so they should be reviewed like build scripts. A YAML deploy pipeline
can mirror `cdt deploy` while keeping paths in `.env`.

```yaml
version: 1
plugins:
  - cdt_steps.offline

pipelines:
  deploy:
    steps:
      - git.prepare_clean_main
      - web.build:
          env: prod
      - web.copy:
          repository: ${WEB_REPOSITORY}
          destination: ${WEB_BUILD_PLACE}
          inner: ${WEB_INNER}
      - web.cache_bust:
          destination: ${WEB_BUILD_PLACE}
          inner: ${WEB_INNER}
      - git.commit_push:
          repository: ${WEB_REPOSITORY}
          message: ${flutter.version}
```

Plugin modules register steps with `@step`:

```python
from cdt.sdk import PipelineContext, step


@step("offline.fetch_config")
def fetch_config(ctx: PipelineContext, output: str):
    path = ctx.project_path(output)
    path.parent.mkdir(parents=True, exist_ok=True)
    path.write_text("{}", encoding="utf-8")
    ctx.values["offline_config_path"] = str(path)
```

Each step receives a `PipelineContext`. Use `ctx.env` for project `.env`
values, `ctx.runner` for commands, `ctx.artifacts` for named build outputs, and
`ctx.values` for lightweight values shared between steps.

```sh
cdt run deploy
cdt pipeline list
cdt pipeline inspect deploy
cdt pipeline inspect deploy --json
cdt pipeline validate
cdt pipeline validate deploy --json
cdt pipeline steps --json
```

Pipeline JSON output is intended for automation and AI agents. It includes
configured pipeline names, plugin modules, registered steps, structured errors,
and, for inspect, the resolved step tree. Parallel groups are supported with an
explicit `parallel: { steps: [...] }` item. See [docs/pipelines.md](docs/pipelines.md)
and [docs/ai-agents.md](docs/ai-agents.md).

YAML Flutter build steps can pass project-specific options such as `flavor`,
`target`, `dart_defines`, `obfuscate`, `split_debug_info`, `no_pub`,
`no_shrink`, and `extra_args`. If omitted, CDT keeps the legacy build defaults.

`cdt prod` Android artifacts are copied to:
- `~/Downloads/app-release.apk`
- `~/Downloads/app-release.aab`

## Tests

```sh
python3 -m venv .venv
.venv/bin/pip install -e '.[dev]'
.venv/bin/python -m pytest -q
```

`.venv/` is ignored by git. Unit tests do not run real Flutter, Xcode,
Firebase, or App Store commands.

For a CLI smoke check:

```sh
.venv/bin/python -m cdt.cli --help
```

## Local install

```sh
pipx uninstall cdt
pipx install -e .
```

Or use helper script:

```sh
chmod +x scripts/reinstall.sh
./scripts/reinstall.sh
```
